# Vela GTM specification

## V — Invariants

- **V1 — LinkedIn SDUI extraction.** When a rendered LinkedIn profile exposes a semantic `Topcard` section and dated `Experience` rows, Vela GTM must recover the visible name, headline, location, and work history without depending on LinkedIn's generated class names.
- **V2 — OpenAI key boundary.** The internal build may store an OpenAI key in the extension's Chrome profile for background-worker calls, but must never expose it to a LinkedIn tab, generated prompt, queue record, or log; production deployment remains server-side.
- **V3 — Recipient verification.** ContactOut `verified` | Email Verifier `valid` | browser-session `confidence_level=high && is_guess=false` → verified. `accept_all`, `unknown`, guessed, LinkedIn-visible, manually entered → unverified but sendable after explicit review. `invalid` | `disposable` → selection ⊥.
- **V4 — Prospect identity.** A normalized LinkedIn `/in/` URL is the preferred queue identity. Spreadsheet-only prospects use a normalized recipient email identity; repeated imports update one prospect instead of creating duplicate outreach rows.
- **V5 — Reviewed Gmail delivery.** Delivery ! explicit final click + reviewed subject/body + syntactically valid visible recipient + exact sender. Provider status visible but non-blocking; `invalid` | `disposable` unselectable. Direct send ! re-verify saved sender authorization. Missing sender → one prefilled Gmail composer per recipient. Single recipient default; multiple ! Settings opt-in. Sender rotation & cross-recipient exposure ⊥.
- **V9 — Central team source.** Supabase is the team-wide source for Gmail account metadata, prospects, and recipient-level delivery activity. Every scheduled, sent, partial, failed, cancelled, manually marked, or historically imported delivery upserts one recipient event; local delivery history remains the offline fallback.
- **V6 — Provider credential boundary.** ContactOut browser cookies remain inside `contactout.com` page; cookie values → storage/messages/logs ⊥. MV3 state stores installation ID + single-use ≤10m reveal approval only. Optional API/OpenAI keys stay background-only; production keys → server.
- **V7 — Credit-bounded contact resolution.** With automatic research enabled, navigating to a new LinkedIn profile authorizes at most one ContactOut browser-session reveal for that profile in the current side-panel session: masked preview → verification poll → reveal → remaining balance metadata. Repeated same-profile refreshes must not spend another credit automatically; the explicit `Refresh` button may retry. Queue approval still names the exact selected count + maximum credits. API token fallback follows Contact Info → People Enrich → broad profile → Email Verifier.
- **V10 — AI-grounded drafting.** After available enrichment, the OpenAI writer receives bounded work context and writes a complete plain-text body plus internal `workNote`; the application supplies the canonical first-touch subject. Prospect claims must come from supplied profile/provider context; sender claims may come from the selected writing guide. Provider context is never permission to invent facts.
- **V11 — Lookup fallback order.** ∀ lookup → ContactOut browser session → ContactOut API? → Apollo? → LinkedIn RSC → rendered Contact Info. Provider success ⊥ later call. LinkedIn result ≠ ContactOut-verified.
- **V8 — Campaign lifecycle.** Campaigns reference normalized prospect identity; add from profile ! persist current personalization note before membership; rename/duplicate/delete ! preserve the underlying prospect records; campaign totals ! contain campaign members only.
- **V12 — Unified sent history.** Analytics and sent history merge Supabase activity with the bounded local ledger and deduplicate the same delivery-recipient event. Historical `Email Sent` values from imported source files become delivery activity, not an export round trip.
- **V32 — Complete, legible data ledgers.** Shared delivery activity pages through the entire deterministically ordered Supabase result rather than truncating at 1,000. Contacts and sent history render at most 100 filtered rows per UI page with accessible navigation and accurate ranges. Imported activity remains `imported`, renders as **Imported from list**, and any clipped history value exposes its complete text through a hover and keyboard-focus tooltip outside table overflow.
- **V33 — Recoverable Apollo discovery.** Research presents the generated audience plan for confirmation before calling Apollo. An exact zero-result People Search receives at most one automatic broader attempt that enables similar titles and removes optional keyword, skill, and industry filters while preserving explicit company and location constraints. A second zero is a successful actionable empty state with explicit edit or title-only retry choices, not a provider error.
- **V34 — Conversational research routing.** The Research composer classifies each turn as `chat` | `plan` | `execute`. Greetings, questions, explanations, brainstorming, ambiguous requests, and non-prospect work remain normal chat. A plan requires explicit people-discovery intent; execution requires both an existing pending plan and explicit confirmation. No ordinary chat turn may call Apollo.
- **V35 — Research-run schema compatibility.** Saving or reading a research run must not fail solely because the additive `total_found` column has not reached the live PostgREST schema cache. The migration adds the column idempotently, and the client retries the legacy run shape on `PGRST204` while preserving the live local match total.
- **V36 — Complete historical imports.** Spreadsheet history with more than one PostgREST response page upserts every eligible recipient event in bounded batches. Re-importing the same list remains idempotent by `client_event_id`; one oversized request or a 1,000-row response ceiling must not silently produce a partial ledger.
- **V37 — Coherent extension runtime protocol.** Before sending a conversational research action, Workspace confirms that the active background worker advertises that exact action. A stale or unknown worker produces a specific reload-and-reopen recovery message, never the generic `Unknown Vela provider action`.
- **V38 — Popup runtime boundary.** Popup extension-only work is gated by its initialized `state.isPreview` runtime capability; no undeclared environment flag may interrupt rewriting, duplicate-recipient checks, or delivery.
- **V39 — Stable discovered-recipient set.** Every syntactically valid address discovered by ContactOut, Apollo, or LinkedIn Contact Info is retained in the canonical contact details with its source and verification state. Changing the active radio selection updates compose context only; it must not remove or reorder another discovered address.
- **V40 — ContactOut identity binding.** ContactOut browser-session preview selects only the response record whose canonical LinkedIn URL, vanity, or member ID matches the requested profile. Any conflicting preview or reveal identity fails closed before an address is saved, shown, or sent.
- **V41 — Complete Gmail catch-up.** A full Gmail sync enumerates every `in:sent` page for every explicitly connected account, then stores every message in those sent threads idempotently by `(gmail_account_id, gmail_message_id)`. A mailbox is complete only with `sync_scope=all_sent_threads`; replies, bounces, sent totals, thread totals, and the latest history cursor must be persisted for incremental sync. Gmail history participates in duplicate-recipient checks, unrelated inbox-only conversations remain outside Supabase, and Google tokens are never persisted.
- **V42 — Sequential US research batches.** Built-in audience prompts explicitly constrain Apollo to the United States. A completed research run with remaining matches exposes a working next-batch control, and `next 100`-style Research messages advance the same plan from Apollo page `n` to `n+1` without replanning or repeating page 1. ContactOut receives only the strong-fit subset, while its credit confirmation states the pulled count, strong count, and skipped no-credit count.
- **V43 — Empty workspace render safety.** Workspace views must render before any research run exists. Research pagination and next-batch state normalize `null` or absent run data to an empty first page instead of interrupting unrelated Dashboard, Scheduled, Contacts, Analytics, or Sent history views.
- **V44 — Complete prospect imports.** Spreadsheet imports upsert every normalized prospect in bounded sequential batches and remain idempotent by `identity_key`. A source list larger than one PostgREST response page must not become a partial Contacts database, even when historical activity is already complete.
- **V45 — Durable import evidence.** Once a prospect payload contains spreadsheet `importEvidence`, later background, bounce, Gmail, or queue upserts that omit that field must preserve it. A newer import may merge or replace evidence explicitly, but an unrelated partial prospect update must not erase source rows, sent timestamps, operators, or touch counts.
- **V46 — Canonical human attribution.** Every canonical outgoing Gmail `initial` or `follow_up` message becomes one sent-history delivery keyed by Gmail message ID. **Sent by** identifies the signed-in Vela operator from the activity actor (including AI research, YOLO, and scheduled execution), while **Mailbox** independently identifies the connected Gmail `From` address. A matching Supabase activity actor outranks Gmail-sync capture metadata during deduplication; the mailbox owner is never inferred to be the human operator.
- **V47 — Stackable LinkedIn-backed approvals.** Apollo enrichment must preserve a valid canonical LinkedIn `/in/` URL before a prospect may enter Approvals, and every approval row with a profile exposes an explicit **View on LinkedIn** action. Ready and approved drafts from every unfinished research run remain stacked together. **Run approved N** sends the already-approved subset; when ready drafts exist, the primary action becomes **Run and approve all**, approves every complete ready draft, includes the already-approved subset, and opens the existing final send confirmation.
- **V48 — Apollo scalar request contract.** Search plans may contain list-valued audience facets, but every Apollo scalar request field is normalized to a string before transport. The planner reserves `company` for at most one exact employer name and routes company categories such as AI infrastructure or colocation into industry or keyword filters.
- **V49 — Canonical first-touch subject.** Every newly composed initial outreach email uses the exact subject **Quick intro + seeking advice**. Templates and AI output control the body only; custom template subjects and per-prospect AI subjects are ignored. Existing sent-history subjects and same-thread follow-ups remain unchanged.
- **V50 — Unified mailbox accounting and follow-up parity.** Dashboard **Sent today**, dashboard mailbox capacity, and the side-panel sender quota use the same deduplicated successful outgoing `initial` | `follow_up` records from Gmail history, Supabase activity, and the local ledger, grouped by the connected `From` mailbox; queue-level `emailSentAt` fallbacks must not inflate mailbox totals. Every accepted initial Gmail send from the side panel, approval review, research run, or YOLO automation carries its selected template's follow-up cadence and templates. A scheduled initial retains that sequence until Gmail sends it, then creates the same reply-aware background follow-up alarms as an immediate initial. Each follow-up preserves Gmail's returned `threadId`, the original subject, and RFC-compliant `In-Reply-To` and `References` headers. Each queued follow-up exposes **Send now**, which runs the same reply check, clears its future alarm, sends through the existing thread, and updates the same local and shared delivery record.
- **V51 — Continuous draft approval.** A successful single-draft **Approve draft** action keeps the review drawer open and immediately loads the next eligible ready draft. If no other ready draft remains, the just-approved draft stays open in its approved state; only an explicit close action, Escape, or completion of the separate send workflow may dismiss the drawer.
- **V52 — Atomic Gmail sender removal.** Removing a connected Gmail sender deactivates its shared account metadata and approved-sender authorization in one authenticated database transaction before deleting the browser copy. A missing, mismatched, or unauthorized shared account must return a specific error; partial backend deactivation and silent zero-row success are forbidden.
- **V53 — Accurate draft triage.** Successful ⌘/Ctrl+Enter approval removes current draft from pending review cohort, advances to next ready draft, and recomputes ordinal against remaining ready drafts. ⌘/Ctrl+Backspace deletes current prospect + campaign membership and advances to same next position; final deletion closes drawer.
- **V54 — Plainspoken outreach.** Writer ! open naturally with `I came across your profile and...`, mention a grounded detail without reciting the prospect's title or position, explain Vela as an early-stage startup building AI agents that help large energy users get powered on faster, and ask one real question. Resume synthesis/praise frames including `I saw your background`, `I'm reaching out because`, `That mix of`, `exactly the kind of experience`, `teams like yours`, `from your side of the table` → retry.
- **V55 — Manual LinkedIn provider order.** Sidebar profile email lookup → ContactOut browser session → ContactOut API? → Apollo? → LinkedIn Contact Info. First verified provider result wins; Apollo people discovery order unchanged.
- **V56 — Mailbox-observed delivery health.** Delivery analytics is keyed by connected Gmail mailbox, never human operator or imported/unattributed activity. A complete baseline contains every sent message and every message in those sent threads; windowed metrics report sent messages, unique sent threads, unique replied threads, Gmail bounce notices, and explicit policy rejects. **Delivered** and recipient spam-folder placement are never inferred. Routine manual and scheduled syncs use silent authorization for the saved mailbox; an account chooser appears only while adding or explicitly reconnecting an account.
- **V57 — Durable workflow completion.** Every successful immediate Gmail send must mark its prospect sent in browser storage and the shared prospect payload before the approval UI settles, so Approvals, Research badges, and reloads cannot restore completed work. Hidden Research-only attention filters never suppress the visible Approvals stack. Outside a ready approval stack, **Draft qualified** navigates to Research and immediately executes a confirmed pending audience plan; without one it focuses the Research composer for the next brief.
- **V58 — Unambiguous Research navigation.** Research is a workspace for creating and running audience plans, not an inbox. Its sidebar entry never displays a prospect count; actionable draft totals belong to Approvals, while run metrics remain inside the active or saved research run.
- **V59 — Durable shared approval removal.** Shared prospect refresh is read-only and may never write the merged browser queue back to Supabase. Explicit queue mutations sync only the changed prospects. Clearing or deleting approvals must receive every requested `identity_key` from an authenticated Supabase delete before removing the local rows; missing DELETE grants/policies and zero-row success are failures, never successful UI state.
- **V60 — Accurate approved-draft navigation.** Opening any complete ready or approved row in Approvals includes the active row in a same-stage drawer cohort and renders a real ordinal such as `1 of N`; an approved row must never appear as `0 remaining`. Approval progression still removes newly approved rows from the pending-ready cohort before selecting the next draft.
- **V61 — Actionable approval delivery.** Dashboard approval sends run the same shared/local duplicate-recipient preflight as the side panel, offer the explicit send-again override required by V28, and pass that decision to the background worker. Cancellation sends nothing. Any other single-send failure names its concrete reason instead of collapsing to `0 sent · 1 need attention`.
- **V62 — Audience-faithful research qualification.** Apollo People Search receives only supported filters: human-readable industry/category labels join the documented keyword query instead of an ignored industry field, and the automatic zero-result fallback preserves exact role titles rather than enabling similar-title expansion. Every fit check receives the user-approved brief and plan alongside profile facts. A completed zero-ready run renders as **No qualified results**, states the actual fit buckets, and never recommends broadening when its fallback already broadened.
- **V28 — Duplicate-recipient gate.** Before immediate or scheduled delivery, Vela checks prior `sent` | `partial` | `scheduled` records in Supabase plus the local ledger. Match → recipient-specific warning + explicit send-again override; no UI path may bypass the background-worker gate.
- **V29 — Vela team authorization.** The extension is gated on first open and requires Supabase Auth with an `@velaenergy.ai` Google identity. Row Level Security enforces membership on every shared table. Google access and refresh tokens → Supabase ⊥.
- **V30 — Approved sender roster.** Sending identities come from the shared `approved_senders` settings table, never client constants. Immediate and scheduled Gmail delivery must reject any mailbox that is not active in that roster. The initial roster is Tony and Tarun.
- **V31 — Service-worker lifecycle safety.** Manifest V3 background-module evaluation only registers listeners and defines helpers. Side-panel behavior and workspace backup mutations run from Chrome lifecycle/storage events; persisted `setPanelBehavior` configuration is installed once and is not relaunched during top-level evaluation or every browser startup.
- **V30 — Contact record.** The Contacts view merges normalized prospects with recipient-level team activity. Imported but unreached people remain visible; sent/partial activity determines touch count and last contact; replies and scheduled sends take distinct status; repeated imports or sends must not create duplicate people.
- **V13 — Credential visibility.** Internal API keys remain masked by default and may be revealed only through an explicit per-field control on the settings page.
- **V14 — Durable scheduled delivery.** Scheduled jobs persist in Chrome storage without OAuth tokens and run through `chrome.alarms`; startup recreates missing alarms. Persistent schedule mode uses the next local occurrence of the chosen time for each explicit send action and remains enabled until the user turns it off.
- **V15 — Reusable writing guides.** Settings stores named body guides with stable IDs, sender identity, calendar URL, and supported merge variables. Selection renders a local fallback immediately, then AI may vary wording, ordering, and paragraph count while preserving the guide's intent and factual sender details; the first-touch subject remains canonical.
- **V16 — ContactOut bridge compatibility.** Private lookup ! execute from `contactout.com/extension/app/` + captured ContactOut client version, never Vela manifest version. Provider and LinkedIn calls ! bounded timeout; first provider failure remains observable.
- **V17 — Native reveal parity + safe diagnostics.** ContactOut encrypted response's canonical profile/member fields → reveal body; preview + reveal stay on `/extension/app/`, client `5.6.18`, source `12`, absent verify job → `null`. Last 150 stage events may persist only allowlisted metadata; cookies, CSRF, keys, UUIDs, names, URLs, emails, phones ⊥.
- **V17 — Persistent profile workspace.** The toolbar action opens a Chrome side panel rather than a transient action popup. The panel remains available while the user changes tabs and refreshes to the active LinkedIn `/in/` profile identity without treating same-profile overlays as a new prospect.
- **V18 — Delivery ledger.** Every accepted Gmail delivery attempt and every scheduled, cancelled, partial, failed, or completed job produces one bounded local record containing sender label, recipients, subject, mode, status, and timestamps. OAuth tokens and provider credentials → ledger ⊥.
- **V19 — Automatic writing and Gmail fallback.** New LinkedIn profile identity → exactly one AI writing attempt regardless of contact research outcome. Any syntactically valid visible | entered address → direct Gmail send | prefilled composer, even when provider verification incomplete.
- **V20 — OAuth client-type routing.** The OAuth client declared in `manifest.json` may authorize Gmail only through `chrome.identity.getAuthToken`; it must never enter `launchWebAuthFlow`. Account-chooser Web OAuth is available only when Settings contains a distinct Web application client ID.
- **V21 — Connected Gmail account set.** User may connect multiple explicit Gmail identities and choose one in compose. Persisted account set contains normalized Google ID/email/auth mode only; OAuth tokens → storage ⊥. Immediate + scheduled delivery ! retain chosen `accountId` + sender label, silently reauthorize that sender with `prompt=none`, re-verify token email before send, and auto-rotation ⊥. Only explicit connect/add/switch actions may use `prompt=select_account`. Legacy single sender migrates without disconnect.
- **V22 — Active compose context.** Selecting valid recipient → address + work/personal/provider/verification metadata promoted into visible email, saved draft, delivery target, writer request. **Rewrite email** produces a new grounded complete draft from that active context and the selected writing guide.
- **V23 — Update-safe workspace.** Queue and campaign collections are mirrored into one versioned local backup. Missing or malformed primary collection keys after startup/update → restore from a valid backup; an explicitly stored empty array remains authoritative and must never resurrect deleted campaigns.
- **V24 — LinkedIn launcher.** Supported LinkedIn profile and people-search pages expose one compact Vela launcher only after the background worker has enabled `popup.html` as that tab's side-panel path. Its user click immediately calls `sidePanel.open({ tabId })` through the background worker without awaiting setup work; unsupported LinkedIn routes show no launcher.
- **V25 — AI-drafted delivery gate.** Every new profile starts one AI drafting attempt while the loading view remains visible. Delivery remains disabled until a grounded complete subject/body/`workNote` succeeds; a missing key, failed request, stale result, or in-flight rewrite must fail closed.
- **V26 — Rich profile grounding.** AI writing receives the visible LinkedIn About copy and bounded descriptions for up to six roles, in addition to titles and companies. Provider enrichment must fill gaps without discarding richer LinkedIn About or role-description text.
- **V27 — Natural complete-draft boundary.** For side-panel and dashboard generation, the model's complete subject/body becomes the reviewed draft. The body uses natural blank-line paragraph separation without hard-wrapped lines, varies structure by prospect, retains exact sender/calendar values when supplied, and contains no unresolved merge variables or invented prospect facts.

## I — External surfaces

- **I1 — LinkedIn people search.** Open a native LinkedIn people-result URL and capture only profile links currently rendered in the signed-in user's browser.
- **I2 — Source spreadsheet import.** Local `.xlsx`/`.xls`/`.csv` import maps common identity, recipient, personalization, message, and historical sent-date columns. Export ⊥.
- **I4 — AI writer.** Existing configured writer endpoint accepts structured profile context, the generation mode, and the current draft.
- **I5 — ContactOut API fallback.** `GET /v1/people/linkedin`, `POST /v1/people/enrich`, `GET /v1/linkedin/enrich`, `GET /v1/email/verify`, `GET /v1/stats`; background-only `token`.
- **I6 — Gmail send.** `POST https://gmail.googleapis.com/gmail/v1/users/me/messages/send` → one base64url RFC 2822 message per reviewed syntactically valid recipient; `gmail.send` authorization.
- **I10 — Manual Gmail handoff.** `https://mail.google.com/mail/?view=cm&fs=1` with URL-encoded `to`, `su`, and `body`; Vela opens the reviewed composer but does not click Gmail's Send control.
- **I7 — Scheduled jobs.** `chrome.storage.local` queue + one-shot `chrome.alarms` entries keyed by job ID; alarm handler sends with the stored Google account ID and records sent or failed state.
- **I8 — ContactOut browser bridge.** MAIN-world `contactout.com` request → `GET /api/user/info`, `POST /api/v5/profiles/encrypted`, `POST /api/email/verify/status`, approved `POST /api/v5/profiles/reveal`; cookies/CSRF stay page-owned.
- **I9 — Chrome side panel.** Manifest V3 `side_panel.default_path` + `sidePanel` permission; background worker calls `chrome.sidePanel.setPanelBehavior({ openPanelOnActionClick: true })` so the toolbar action opens Vela beside the active page.
- **I11 — Google authorization.** Google Web application client → `chrome.identity.launchWebAuthFlow` with `prompt=select_account` → exact `https://<extension-id>.chromiumapp.org/google` redirect → transient ID token + `gmail.send`/`userinfo.email` access token. The ID token establishes a Supabase session; persisted sender state contains account IDs/emails/auth modes only; Google token storage ⊥.
- **I12 — Supabase team data.** Auth token exchange plus PostgREST access to RLS-protected `team_profiles`, `gmail_accounts`, `prospects`, and `activity_events`. The extension uses the publishable key; service-role credentials and Google tokens → client/database ⊥.
- **I13 — Google Sheets source import.** Sheets have no runtime API integration. Users download `.xlsx`/`.csv`; local mapping and AI-assisted normalization convert rows to shared prospects and historical delivery activity. Export ⊥.

## B — Bug history

- **B1 · 2026-07-13 · V1.** The extractor assumed a `main h1` and `span[aria-hidden='true']` nodes. LinkedIn's current SDUI profile renders its top card with an `h2`, ordinary text nodes, and opaque classes, leaving headline, location, and experience empty even though they were visible.
- **B2 · 2026-07-13 · V3.** Email extraction only inspected the base profile DOM, while LinkedIn exposes the address after its `ProfileContactDetailsOverlay` navigation renders a `mailto:` link.
- **B3 · 2026-07-13 · V11.** Verified-only popup refactor returned on ContactOut miss or error → LinkedIn fallback never ran.
- **B4 · 2026-07-13 · V16.** Session check accepted Vela `0.6.0` on `/login/callback`, but private lookup required ContactOut's extension-app context/client contract; popup hid the bridge failure behind API sample/Apollo fallbacks and an unbounded LinkedIn wait.
- **B5 · 2026-07-13 · V17.** Native ContactOut preview accepted member `0`, resolved the canonical member ID, then revealed with that returned descriptor on `/extension/app/`; Vela retained member `0` and could switch reveal back to the login tab, with no safe stage log to expose the divergence.
- **B5 · 2026-07-13 · V5.** Gmail setup depended on `chrome.identity.getAccounts()`, leaving supported Chrome installs stuck on “Update Chrome”; connection now authorizes and labels the primary Chrome profile through stable Identity APIs and requests only `gmail.send`.
- **B6 · 2026-07-13 · V7,V10.** The side panel asked for a second credit confirmation after the user already clicked `Find verified`, while AI output could return a third-person phrase such as `Jonathan's progression…` that became awkward inside `impressed by {{workNote}}`. The manual lookup click now authorizes one reveal, remaining credits render as metadata, and writer output is prompted and normalized for the exact template slot.
- **B7 · 2026-07-13 · V7,V10.** New LinkedIn profiles stopped at a ready state and still required `Find verified`, while the default email read as disconnected praise, company boilerplate, and ask blocks. New profile identities now start one guarded enrichment/personalization run automatically, and untouched legacy quick-intro templates migrate to a tighter founder intro → specific reason → call request structure.
- **B8 · 2026-07-13 · V5.** Stable Chrome's `identity.getAuthToken()` silently chose the profile's sync/first Google account and `identity.getAccounts()` remained Dev-channel-only, so an external primary account could hit `org_internal` before any chooser appeared. Google delivery now uses a Web OAuth client with `select_account`, verifies the returned email, and binds future sends to that selected sender.
- **B9 · 2026-07-13 · V19.** Disconnected Gmail reused the direct-send verified-recipient list, leaving **Open Gmail** disabled for valid visible addresses, while profile startup tied AI writing to successful automatic contact enrichment. Manual compose now accepts a valid current address without weakening direct-send verification, every new profile starts one AI write, and redirect mismatch errors name the exact Web-client URI Google Cloud must authorize.
- **B10 · 2026-07-13 · V20.** Settings accepted the manifest's Chrome-extension client ID as if it were a Web application client and passed it to `launchWebAuthFlow`, so Google deterministically returned `redirect_uri_mismatch`. OAuth strategy selection now routes that client through `getAuthToken` and permits the redirect flow only for a distinct optional Web client.
- **B11 · 2026-07-13 · V22.** Recipient radio changes updated only `selectedRecipients`, leaving the visible email and writer context on the prior address. Selection now promotes one active, provider-classified recipient across compose, persistence, writing, and delivery.
- **B12 · 2026-07-13 · V22.** The button labeled **Rewrite with AI** sent `generationMode=personalization`, instructed the model to return the existing subject/body verbatim, and then rebuilt that same template. Explicit rewrites now use full mode with the current draft and complete compose context, while automatic writing keeps the configured conservative mode.
- **B13 · 2026-07-13 · V23.** Campaigns existed under one bare local-storage key, so an update/install identity transition or missing key could make the dashboard silently render an empty campaign list. The pinned extension ID prevents future namespace drift, and a versioned workspace backup now repairs missing primary collections without undoing intentional deletion.
- **B14 · 2026-07-13 · V25.** Profile loading revealed the workspace before automatic writing began, and delivery readiness accepted any non-empty template subject/body.
